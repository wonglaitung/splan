### 1.1 规划书核心主张

## 二、统一AI能力接入网关（安全层）

> **对应规划书章节**：三、接入网关的三大硬核控险机制

### 2.1 核心价值

| 价值维度 | 具体收益 | 商业意义 |
|---------|---------|---------|
| **合规保障** | 从源头免疫隐私泄露红线，满足《个人信息保护法》 | 避免监管处罚、声誉风险 |
| **风险防控** | 技术护栏代偿行政管理组织的缺失，提供刚性安全承诺 | 在无AI治理委员会过渡期安全落地 |
| **效率提升** | 统一入口，业务部门无需各自开发安全模块 | 降低重复建设成本 |
| **算力集约** | 集中式处理，避免各应用重复部署安全检测模型 | 节省GPU资源 |

### 2.2 架构亮点：双层护栏设计

```
请求流程：
┌──────────┐    ┌────────────────┐    ┌────────────────┐    ┌──────────┐
│ 业务系统 │───▶│ Layer 1 检测   │───▶│ Layer 2 Judge  │───▶│ 上游模型 │
│          │    │ (PII规则检测)  │    │ (意图裁判)     │    │          │
└──────────┘    └────────────────┘    └────────────────┘    └──────────┘
                      │                     │
                      ▼                     ▼
               ┌────────────┐        ┌────────────┐
               │ PII脱敏    │        │ 风险检测   │
               │ 如：<手机号>│        │ 高风险阻断 │
               └────────────┘        └────────────┘
```

| 护栏层 | 技术方案 | 响应时间 | 检测内容 |
|-------|---------|---------|---------|
| **Layer 1** | 正则 + Microsoft Presidio NER | < 1ms | 身份证、手机号、银行卡号、护照号、统一社会信用代码等 12 类 PII |
| **Layer 2** | Qwen3Guard-8B-Stream LLM Judge | ~100ms | Prompt注入检测、越权意图检测、风险等级判定 |

**Layer 1 支持的 PII 类型**：

| 实体类型 | 示例 | 脱敏占位符 |
|---------|------|-----------|
| 手机号 | 13812345678 | `<手机号>` |
| 身份证 | 110101199001011234 | `<身份证号>` |
| 银行卡 | 6222021234567890123 | `<银行卡号>` |
| 护照 | E12345678 | `<护照号>` |
| 统一社会信用代码 | 91110000600007333L | `<统一社会信用代码>` |

### 2.3 与规划书三大硬核控险机制的对应

| 规划书要求 | 技术实现 | 实现状态 |
|-----------|---------|---------|
| **隐私数据动态脱敏机制** | Layer 1 PII识别器（正则 + NER），实时脱敏 | ✅ 已完整实现 |
| **IAM权限强制继承机制** | IAM集成模块，权限标签注入RAG检索 | 🔴 **关键缺口，待实现** |
| **输入实时过滤护栏** | Layer 2 Judge 检测 prompt_injection | ✓ 已实现 |
| **输出实时过滤护栏** | SSEHandler + StreamInterceptor Token级拦截 | ✓ 已实现框架 |

### 2.4 关键设计：解耦式护栏架构

**核心原则**：边缘治理、算力解耦。

```
❌ 错误路径（耦合架构）：
┌─────────────────────────────────────────────────┐
│           vLLM 推理引擎容器                      │
│  ┌─────────────┐  ┌─────────────┐              │
│  │ 模型推理    │  │ 安全Handler │ ← 资源竞争！  │
│  │ (GPU算力)   │  │ (嵌入内部)  │              │
│  └─────────────┘  └─────────────┘              │
└─────────────────────────────────────────────────┘
问题：安全检测拖慢GPU吞吐率30-50%

✓ 正确路径（解耦架构）：
┌────────────────────┐     HTTP/gRPC     ┌────────────────────┐
│   安全网关容器     │ ────────────────▶ │  推理引擎容器      │
│ ┌────────────────┐ │     异步通信      │ ┌────────────────┐ │
│ │ PII检测        │ │                  │ │ 纯粹推理       │ │
│ │ Judge裁判      │ │                  │ │ (GPU满载)      │ │
│ │ 流式拦截       │ │                  │ └────────────────┘ │
│ └────────────────┘ │                  │     CPU/内存独立   │
│     CPU密集型      │                  └────────────────────┘
└────────────────────┘                        GPU 硬绑定
```

**收益**：
- GPU 推理集群 100% 专注算力满载
- 网关更新不影响推理引擎（显存常驻）
- 故障隔离：单点故障不扩散

## 三、研发智能助手（效能层）

> **对应规划书章节**：四、Run（卓越运营领域）场景A

### 3.1 核心价值

| 价值维度 | 具体收益 | 规划书目标 |
|---------|---------|-----------|
| **科技自我造血** | 先用AI帮科技团队把简单修补代码工作消化掉 | 研发综合提效 35% 以上 |
| **填补知识断层** | 解决"AI不认识内部框架"问题 | 避免"凭通用知识猜测内部框架行为" |
| **释放核心人力** | 腾出人力承接业务条线更复杂的智能化急单 | 实现团队自我造血 |

### 3.2 核心问题：通用 Claude Code 的能力边界

**Claude Code 的强项**：
- ✅ 通用编程知识（Python、Java、Go、SQL）
- ✅ 常见框架（Spring Boot、Django、FastAPI、React）
- ✅ 标准 API（REST、gRPC、OAuth）

**Claude Code 的弱项**：
- ❌ 企业内部框架（如银行核心系统接口）
- ❌ 私有库和定制组件
- ❌ 业务特定规范（如行内编码规范、安全守则）

**真实案例**（来自实际开发教训）：

> 修复"代理 API 不支持 tool role"问题时，AI 凭假设修改了代码：
> - **假设**：Anthropic API 有 `role: "tool"` 这个角色（因为 OpenAI API 有）
> - **实际**：Anthropic API 根本没有 tool role，工具结果是 user message
> - **结果**：三次修改才找到正确方案
>
> **根本原因**：AI 凭训练数据中的通用知识"猜测"内部框架行为，而不是查阅官方文档。

### 3.3 架构亮点：内网知识库注入

**示例：财富管理多 Agent 场景**：

```java
@Service
public class WealthOrchestrator {

    @Autowired
    private AgentService marketMonitorAgent;

    @Autowired
    private AgentService clientAdvisorAgent;

    @Autowired
    private AgentService notificationAgent;

    public WorkflowResult executeMarketAlert(String event) {
        // 1. 市场 Agent 检测异动
        LoopResult analysis = marketMonitorAgent.run(
            "分析市场异动：" + event
        );

        // 2. 决策 Agent 筛选客户
        LoopResult clients = clientAdvisorAgent.run(
            "筛选受影响的客户：" + analysis.content()
        );

        // 3. 通知 Agent 生成建议
        LoopResult notification = notificationAgent.run(
            "为以下客户生成通知：" + clients.content()
        );

        return WorkflowResult.completed(notification);
    }
}
```

## 六、关键设计决策解读

### 6.1 决策一：采用解耦式架构

**问题背景**：

在银行内网环境中，网络栈配置复杂，防火墙规则严格。Docker Compose 等微服务编排工具可能产生隐形不确定性：

| 风险 | 说明 |
|-----|------|
| 配置解析失败 | Docker Compose 的 YAML 解析在特定环境下可能失败 |
| 启动顺序依赖 | 编排工具的依赖解析可能导致启动卡死 |
| 网络栈冲突 | overlay 网络与行内网络策略冲突 |
| 故障扩散 | 单容器故障可能影响整个编排链 |

**解决方案**：

采用 **原生 Docker Run** 模式，单容器独立运行：

```bash
# 推理引擎（独立启动）
docker run -d --name inference-engine --gpus device=0 ...

# 安全网关（独立启动）
docker run -d --name guardrails-gateway ...

# 知识库服务（独立启动）
docker run -d --name mcp-api-service ...
```

**收益**：

| 收益 | 说明 |
|-----|------|
| 故障边界清晰 | 单容器故障不影响其他服务 |
| 网关更新不影响推理引擎 | 推理引擎可保持长连接、显存不释放 |
| 显存常驻 | 推理引擎持续运行，无需重复加载模型（加载72B模型需15-30分钟） |

### 6.2 决策二：网关与推理引擎必须隔离

**问题背景**：

曾有技术尝试将"防幻觉"或"内容审核"直接写入 vLLM 等推理引擎的自定义 Handler 内部。

| 缺陷路径 | 问题说明 | 影响 |
|---------|---------|------|
| 安全逻辑嵌入推理引擎 | 成倍拖慢底层算力的 Token 吞吐率（TTFT） | 性能下降 30-50% |
| 框架耦合 | 推理引擎升级频繁，安全模块极难维护 | 维护成本激增 |
| 资源竞争 | 安全检测与推理共享 GPU 显存 | 显存碎片化，吞吐不稳定 |

**解决方案**：

采用 **边缘治理、算力解耦** 设计：

- 安全检测逻辑在网关层独立运行
- 网关通过 HTTP/gRPC 与推理引擎通信
- GPU 推理集群 100% 专注算力满载

**收益**：

- 零服务闪断：网关高频更新不影响推理引擎
- 显存常驻：推理引擎持续运行，无需重复加载模型
- 故障隔离：网关崩溃不影响推理引擎

### 6.3 决策三：技术护栏替代行政审批

**问题背景**：

行内暂无跨部门的 AI 伦理、数据安全与风控治理委员会。科技部门在推进应用时，需独自承担 AI 生成内容的合规与内容风险兜底责任。

**解决方案**：

在网关层硬编码安全规则，实现"技术先于行政，风险技术一票否决"：

| 技术护栏 | 功能 | 触发条件 |
|---------|------|---------|
| PII 动态脱敏 | 身份证、手机号、银行卡号自动遮掩 | 所有请求 |
| Prompt 注入检测 | 拦截"忽略之前的指令"等攻击 | 检测到注入模式 |
| 越权意图检测 | 识别"你现在是管理员"等越权尝试 | LLM Judge 判定 |
| 幻觉检测 | 引用溯源验证、置信度评分 | 输出阶段 |
| 合规规则库 | 拦截"保本保息"等违规表述 | 输出阶段 |

**收益**：

- 提供刚性的安全承诺
- 在过渡期安全落地
- 为后续建立 AI 治理委员会赢得时间

### 7.1 总体支撑能力评估

| **业务场景** | ★★★★☆ | Run/Protect场景支撑充分，Grow场景需增强 |
| **技术可行性** | ★★★★★ | 解耦架构、技术选型均符合银行内网约束 |
| **成熟度演进** | ★★★★☆ | L1-L3路径清晰，L4-L5需进一步规划 |

### 7.2 三大硬核控险机制实现状态

| 规划书机制 | 实现状态 | 风险等级 | 说明 |
|-----------|---------|---------|------|
| **隐私数据动态脱敏** | ✅ 已完整实现 | 🟢 低风险 | 12类PII识别器覆盖全面，实时脱敏 |
| **IAM权限强制继承** | ⚠️ 框架预留，待实现 | 🔴 高风险 | 规划书核心要求，当前缺失 |
| **输入/输出双向过滤** | ✅ 已实现 | 🟢 低风险 | Layer 2 Judge + 合规规则库 |

**IAM权限继承缺口详解**：

规划书原文强调：
> "低权限员工绝对无法通过向大模型进行'提示词诱导提问'，越权获取高密级的内部文档。"

当前技术实现的局限性：

| 层面 | 当前状态 | 风险 |
|-----|---------|------|
| 身份识别 | 未对接行内LDAP/AD | 无法获取用户真实权限 |
| 权限注入 | 框架预留接口 | 无实际权限数据流转 |
| RAG过滤 | 无权限标签机制 | 检索结果无法按权限过滤 |
| 越权检测 | Layer 2 Judge可检测意图 | 仅能告警，无法真正阻止访问 |

**解决方案路径**：

```
┌─────────────────────────────────────────────────────────────────────────┐
│                   IAM权限继承模块（待实现）                               │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │ 1. 用户身份识别                                                  │   │
│  │    - 从请求头提取用户 Token/Session                              │   │
│  │    - 调用行内 IAM 服务（LDAP/AD）获取用户权限列表                │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │ 2. 权限标签注入                                                  │   │
│  │    - 将权限信息注入到请求上下文                                   │   │
│  │    - 定义权限等级：public → internal → confidential → secret     │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │ 3. RAG检索过滤                                                   │   │
│  │    - 知识库文档标注权限等级                                       │   │
│  │    - 检索时按用户权限过滤结果                                     │   │
│  │    - 网关层拦截无权限的检索请求                                   │   │
│  └─────────────────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────────────────┘
```
